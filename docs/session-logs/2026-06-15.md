## Existing Application Data

The intended stored job record contains:

```text
id
company
role
location
dateApplied
status
link
notes
```

### Field Findings

- `id`
  - Starter records use numeric IDs.
  - New records use `Date.now()`.
  - JobTrackr Pro should reconsider whether IDs should be strings for easier future backend compatibility.
- `company`
  - Used by the form, cards, and search.
  - Should be retained.
- `role`
  - Used by the form, cards, and search.
  - Should be retained.
- `location`
  - Stored as one free-text value.
  - Should be retained for the MVP.
- `dateApplied`
  - Starter records use values such as `17/4/2026`.
  - The HTML date input produces values such as `2026-04-17`.
  - The mixed date formats should not be migrated unchanged.
  - JobTrackr Pro should store one consistent machine-readable format and format it only for display.
- `status`
  - Existing values are `Applied`, `Interviewing`, `Offer`, and `Rejected`.
  - These values should become a TypeScript union or derive from a typed constant.
- `link`
  - Included in the creation form.
  - Not present in the starter records.
  - Not displayed in the current cards.
  - Should be retained as an optional field and surfaced on the details page.
- `notes`
  - Included in the creation form.
  - Not present in the starter records.
  - Not displayed in the current cards.
  - Should be retained as an optional field and surfaced on the details page.

## Starter Data Findings

- Three starter applications are declared directly inside `App.jsx`.
- Starter data is used only when there is no saved localStorage value or saved JSON cannot be parsed.
- Starter records do not have the same complete shape as form-created records because they omit `link` and `notes`.
- JobTrackr Pro should:
  - Move starter data into a dedicated typed data file.
  - Give every starter record the same valid shape.
  - Correct spelling and presentation inconsistencies.
  - Use a consistent date format.
  - Keep starter data separate from component code.

## Existing localStorage Flow

The original flow is:

1. Read the `jobs` localStorage key during the initial `useState` call.
2. Use starter jobs when the key does not exist.
3. Parse saved JSON inside `try/catch`.
4. Fall back to starter jobs when JSON parsing fails.
5. Save the complete jobs array whenever state changes.

### What Is Worth Retaining

- Lazy initial loading.
- Starter-data fallback.
- JSON parse protection with `try/catch`.
- Persisting after application state changes.
- Keeping one application array as the source of truth.

### What Should Improve

- The key `jobs` is too generic.
  - Use a namespaced key such as `jobtrackr-pro:applications`.
- Successful JSON parsing does not prove that the value is a valid application array.
  - Add basic runtime validation before trusting stored data.
- Direct localStorage access cannot run in a Next.js Server Component.
  - Browser storage logic must live behind a Client Component boundary.
- Storage concerns should not remain inside a large page component.
  - Move read/write behavior into focused utilities and later expose application actions through `useApplications`.
- Storage failures during `setItem` are not handled.
  - JobTrackr Pro should decide how to report or recover from write failures.
- The initial hydration strategy needs care in Next.js.
  - Avoid reading localStorage during server rendering.
  - Decide how to represent the short loading period before browser data is available.

`App.jsx` currently owns:

- The complete jobs array.
- localStorage initialization.
- localStorage persistence.
- Search text.
- Status filter.
- Add behavior.
- Delete behavior.

The flow is:

```text
App state
  -> StatsCards
  -> SearchBar
  -> JobForm
  -> JobList

```

### Retain

- One source of truth for the application array.
- Derived statistics rather than separate statistics state.
- Derived filtered results rather than a second stored array.
- Controlled form inputs.
- Functional state updates when adding and deleting.

### Redesign

- Do not recreate one oversized Next.js page that owns everything.
- Centralize application data behavior in a custom hook when the required API is understood.
- Let routes and feature components consume the shared application behavior.
- Keep search and filter state close to the applications list unless another route genuinely needs it.
- Keep temporary form input state inside the form component.

## Existing Workflows

### Create

- The form collects company, role, location, date applied, status, link, and notes.
- Submission prevents the browser refresh.
- The form sends its current data to the parent.
- The parent adds an ID and appends the record.
- The form resets after submission.

Migration decision:

- Retain the controlled-form concept.
- Move creation to `/applications/new`.
- Add TypeScript input types and validation.
- Redirect or navigate after a successful submission.

### Read

- Applications are displayed as cards in one list.
- Cards show role, company, location, status, and date applied.
- Link and notes are not displayed.

Migration decision:

- Retain a reusable list/card concept.
- Use `/applications` for the collection.
- Use `/applications/[id]` for the complete record, including notes and job link.

### Update

- Edit behavior does not exist in the original MVP.
- Status cannot be updated directly.

Migration decision:

- Build update behavior as a new JobTrackr Pro feature.
- Use `/applications/[id]/edit`.
- Reuse the application form for both create and edit modes where practical.

### Delete

- Delete filters the selected numeric ID out of the jobs array.
- Deletion happens immediately without confirmation or undo.

Migration decision:

- Retain ID-based deletion.
- Add a deliberate confirmation pattern before destructive deletion.
- Decide whether deletion belongs on the details page, list card, or both.

### Search and Filter

- Search matches role or company, case-insensitively.
- Status filtering supports all four statuses plus `All statuses`.
- Search and status conditions are combined.
- An empty-state message appears when no records match.

Migration decision:

- Retain all current behavior.
- Type the filter state.
- Derive options from the same central status definition used by forms and analytics.
- Consider adding location search later, but do not expand MVP scope yet.

### Statistics

- Statistics are derived from the current jobs array.
- The original dashboard shows:
  - Total jobs.
  - Interviewing.
  - Offers.
  - Rejected.

Migration decision:

- Retain derived statistics.
- Move the overview to `/`.
- Reuse the same source data for `/analytics`.
- Avoid storing totals independently.

## Proposed JobTrackr Pro Data Direction

The next session should decide the exact definitions, but the likely model is:

```text
ApplicationStatus
  "Applied" | "Interviewing" | "Offer" | "Rejected"

JobApplication
  id
  company
  role
  location
  status
  dateApplied
  link
  notes

JobApplicationInput
  the editable form fields without the generated id
```

Questions to resolve next:

- Should `id` be a string?
- Should `link` and `notes` be optional properties or required strings that may be empty?
- Should the status labels remain title-cased values or use stable lowercase values with separate display labels?
- Should timestamps such as `createdAt` and `updatedAt` be introduced now or deferred?
- What exact ISO date representation should be stored?
- How much runtime validation is appropriate for localStorage data in this MVP?
