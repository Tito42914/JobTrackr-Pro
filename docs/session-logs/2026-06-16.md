## Type File Created

Created:

```text
src/types/application.ts
```

This file defines the core application data contract.

### `ApplicationStatus`

Defines the only valid internal status values:

```text
applied
interviewing
offer
rejected
```

Purpose:

- Prevent accidental invalid status values.
- Provide a shared status type for forms, filters, cards, analytics, and storage.
- Keep internal values stable and simple.

### `JobApplication`

Defines a complete stored job application record.

Fields:

- `id`
- `company`
- `role`
- `location`
- `status`
- `dateApplied`
- `link`
- `notes`
- `createdAt`
- `updatedAt`

Purpose:

- Give all application records one reliable shape.
- Improve on the original JavaScript object structure from the Vite project.
- Prepare the data model for future localStorage, API, and database use.

### `JobApplicationInput`

Defines the shape of form data before the app adds generated fields.

It omits:

- `id`
- `createdAt`
- `updatedAt`

Purpose:

- Separate user-entered data from app-generated metadata.
- Make create and edit forms easier to type later.
- Avoid letting forms manually control fields that should be generated by the application.

## Status Constants Created

Created:

```text
src/constants/applicationStatuses.ts
```

This file contains:

- `APPLICATION_STATUSES`
- `APPLICATION_STATUS_LABELS`

### `APPLICATION_STATUSES`

Purpose:

- Provide a central ordered list of valid statuses.
- Support future dropdowns, filters, analytics, and reusable UI.
- Avoid manually repeating status strings throughout the app.

### `APPLICATION_STATUS_LABELS`

Purpose:

- Map internal lowercase status values to user-facing labels.
- Example: `applied` can be displayed as `Applied`.
- Keep data values separate from presentation text.

The `Record<ApplicationStatus, string>` type means:

- Every valid status must have a corresponding label.
- Missing labels are caught by TypeScript.
- Invalid status keys are caught by TypeScript.
