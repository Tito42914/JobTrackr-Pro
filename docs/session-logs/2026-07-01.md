## Work Completed

### Added localStorage persistence

Updated:

```text
src/features/applications/hooks/useApplications.ts
```

The hook now:

- Reads stored applications from localStorage when running in the browser.
- Falls back to `starterApplications` if no saved data exists.
- Falls back to `starterApplications` if stored data cannot be parsed.
- Saves the current applications array back to localStorage whenever it changes.

Learning point:

- `localStorage` is browser-only, so the hook needs a `typeof window === "undefined"` guard.
- This protects the app from trying to read browser APIs during server rendering.

### Resolved the hydration mismatch

Updated:

```text
src/features/applications/providers/ApplicationsProvider.tsx
```

The provider now uses `useSyncExternalStore` to create a client-safe rendering boundary.

Why this was needed:

- The server cannot read localStorage.
- The browser can read localStorage.
- If the server renders starter data but the browser immediately renders saved localStorage data, React detects different HTML and raises a hydration error.

The fix:

- The provider returns `null` while React is matching the server-rendered output.
- Once safely on the client, it renders the inner provider and allows `useApplications` to load browser data.

Learning point:

- Hydration is React connecting browser interactivity to server-rendered HTML.
- A hydration mismatch means the HTML React expected from the server did not match what the client tried to render first.
- This was not a broken save; it was a server/client rendering timing issue.

## Current Architecture Snapshot

The main application data flow is now:

```text
localStorage / starterApplications
-> useApplications
-> ApplicationsProvider
-> useApplicationsContext
-> ApplicationsList / NewApplicationForm / EditApplicationForm / ApplicationDetails
```

The route structure involved is:

```text
src/app/applications/layout.tsx
src/app/applications/page.tsx
src/app/applications/new/page.tsx
src/app/applications/[id]/page.tsx
src/app/applications/[id]/edit/page.tsx
```

The key idea:

- Route files decide what page exists.
- Feature components decide what each page does.
- The hook owns application state behaviour.
- The provider shares that state across the route tree.
